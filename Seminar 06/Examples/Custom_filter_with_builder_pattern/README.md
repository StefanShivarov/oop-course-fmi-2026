# Builder Design Pattern

### ℹ️ [Builder - допълнителна информация](https://refactoring.guru/design-patterns/builder)

**Builder** е creational design pattern, който се използва за създаване на обекти стъпка по стъпка.  
Той е особено полезен, когато:

- класът има много параметри
- искаме да си спестим излишно дълги конструктори с много аргументи
- има много **незадължителни**(optional) полета
- има много различни възможни конфигурации на полетата в класа
- или когато искаме създаването на обекта да бъде по-четимо и по-ясно

Вместо да създаваме обект чрез дълъг и труден за четене конструктор, използваме специален **builder**, който постепенно конфигурира желаните стойности и накрая създава крайния обект.


## Какъв проблем решава

Нека си представим клас с много възможни настройки.  
Ако използваме обикновен конструктор, често се стига до нещо подобно:

```cpp
Config conf(true, 10, false, 0, true, 25, false, nullptr);
```

Такъв код е труден за четене, защото не се вижда ясно какво означава всяка стойност. Освен това сме длъжни да подадем всички аргументи в конструктора, дори и да ни трябват само някои от тях.

**Builder pattern** решава този проблем, като позволява създаването на обекта да стане по-описателно и в същото време премахва нуждата от това да изброяваме всичко останали аргументи с неутрални стойности:

```cpp
Config c = Config::builder()
    .withValue(10)
    .withLimit(25)
    .build();
```

Тук кодът е много по-разбираем, кратък и по-лесен за поддръжка.


## Основна идея

Идеята е да отделим **конфигурирането** на обекта от **реалното създаване** на обекта.

Builder-ът пази временна информация за желаната конфигурация.  
След като всички нужни параметри са зададени, чрез метод като `build()` се създава крайният обект.



## Типичен начин на използване

```cpp
ProductFilter customFilter = ProductFilter::builder()
    .withMaxPrice(2300)
    .withMinQuantity(3)
    .build();
```

В този пример имаме клас, който играе ролята на филтър за обекти от тип `Product`. Всяко от полетата на филтъра определя някакъв критерий за филтриране, като не е задължително всички от тях да бъдат посочени, тоест може да се филтрира само по някои от тези критерии.

- `ProductFilter::builder()` създава builder, с първоначално зададени неутрални стойности на всички полета
- `withMaxPrice(2300)` задава максимална цена
- `withMinQuantity(3)` задава минимално количество
- `build()` връща готовия обект


## Как обикновено се имплементира

Често класът съдържа вложен `Builder` клас и статичен метод `builder()`, който връща обект от този builder.

Типична структура:

```cpp
class Config {
public:
    class Builder;

    static Builder builder();
};
```

След това вложеният `Builder` клас пази всички временни настройки и предоставя методи за тяхното задаване:

```cpp
class Config::Builder {
public:
    Builder& withValue(int value);
    Builder& withLimit(int limit);

    Config build() const;
};
```

Накрая `build()` създава и връща крайния обект.

---

### Защо builder методите връщат `Builder&`

Методи като:

```cpp
Builder& withLimit(unsigned limit);
```

обикновено връщат референция към текущия builder, за да може извикванията да се **chain-ват**:

```cpp
Config::builder()
    .withValue(10)
    .withLimit(20)
    .build();
```

Идеята е да могат да се chain-ват по този начин, и наподобяват нещо като "конвейерна лента" във фабрика, върху която вървят някакви предмети и им се наслагват отделните елементи един по един.

---

### Каква е идеята на `build()`

Методът `build()` има една основна задача - да създаде крайния обект на базата на всички предварително зададени параметри.

Това е и подходящото място за валидация.

Например, ако дадени параметри са в противоречие, `build()` може да откаже създаването на невалиден обект.

---

### Предимства:

- #### По-четим код

    Вместо поредица от стойности, които трудно се разчитат, имаме ясно именувани стъпки.

- #### По-голяма гъвкавост

    Можем да задаваме само тези параметри, които ни трябват.

- #### По-лесно разширяване

    Ако към класа добавим нова настройка, често е достатъчно да добавим нов метод в builder-а, без да променяме начина на използване на останалия код.

- #### По-добър контрол върху създаването

    Можем да валидираме входните данни преди крайният обект да бъде създаден.


### Недостатъци:

- #### Повече код

    Builder pattern обикновено изисква допълнителен клас и повече boilerplate код.

---

### Кога не е добра идея да го използваме

Не е необходимо да използваме Builder pattern за всеки клас.  
Ако обектът се създава с 1–2 прости параметъра и няма сложна конфигурация, обикновен конструктор е напълно достатъчен.

Например:

```cpp
Point p(3, 4);
```

Тук например builder pattern би направил кода излишно сложен.
