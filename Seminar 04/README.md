# 04. Процес на компилация и линкване (build process). Разделна компилация. Статични променливи и функции. Статични член-данни и член-функции.

Досега разглеждахме предимно по-малки програми, написани в един файл. Това е удобно в началото, но при по-големи програми бързо започват да се появяват проблеми:

- кодът става труден за четене и поддръжка;
- различни части на програмата се смесват на едно място;
- става трудно няколко човека да работят едновременно;
- малка промяна в една част от кода води до прекомпилиране на твърде много неща;
- трудно се разделя интерфейсът от реализацията.

---

## Как от код стигаме до изпълним файл (build process)

Най-общо процесът изглежда така:

1. **Препроцесор**
2. **Компилация**
3. **Обектни файлове**
4. **Линкване (linking)**
5. **Изпълним файл / библиотека**

Важно е да разграничаваме тези етапи, защото различните грешки възникват на различни места.

---

## Препроцесор

Преди същинската компилация изходният файл се обработва от **препроцесора**. Той изпълнява всички **директиви**, които започват с `#`.

Най-често срещаните са:

- `#include`
- `#define`
- `#ifndef`
- `#ifdef`
- `#pragma once`

Важно: препроцесорът **не разбира C++ на ниво класове, функции и типове**. Той работи основно върху текста на файла.

### `#include`

Когато препроцесорът срещне:

```cpp
#include "Student.h"
```

той заменя тази директива със съдържанието на съответния файл.

След това и включеното съдържание също минава през препроцесора. Това означава, че `#include` може да доведе до нови `#include`-и и така нататък.

### `#define`

`#define` се използва за дефиниране на **макроси**.

```cpp
#define PI 3.14159
```

```cpp
double area = PI * r * r;
```

Можем да правим макроси и за "функции":

```cpp
#define SQUARE(x) x * x
```

На пръв поглед изглежда удобно, но трябва да помним, че това е **текстова замяна**.

```cpp
int result = SQUARE(1 + 2);
```

След замяната получаваме:

```cpp
int result = 1 + 2 * 1 + 2;
```

което не е същото като:

```cpp
int result = (1 + 2) * (1 + 2);
```

Затова с макросите трябва да се работи внимателно.

Препроцесорът подготвя изходния код за компилатора, като обработва препроцесорните директиви (#include, #define, #ifdef и др.) и генерира разширен текст на програмата, който след това се подава на компилатора.

## Компилация

След препроцесора идва същинската **компилация**.

Компилаторът:
1. проверява дали кодът спазва правилата на езика
2. открива синтактични и семантични грешки
3. превежда кода до ниско ниво (assembly / machine code)
4. създава **обектен файл** (.obj)

### Синтактични грешки

Това са грешки, при които кодът не е написан според граматиката на езика.

```cpp
int main() {
    int x = 5
}
```

Липсва `;`.

### Семантични грешки

Тук кодът може да е синтактично валиден, но не е смислен според правилата на C++.

```cpp
int main() {
    int x = "hello";
}
```

Друг пример:

```cpp
class Test {
private:
    int x = 10;
};

int main() {
    Test t;
    // t.x = 5; // грешка: x е private
}
```

---

## Обектни файлове

Всеки `.cpp` файл се компилира **самостоятелно** и в резултат се получава **обектен файл**:

- `.obj` при MSVC / Visual Studio;
- `.o` при GCC / Clang.

Обектният файл съдържа машинен код и допълнителна информация за символите, която е нужна на линкера.

Тук е важно да разберем една ключова идея:

Всеки `.cpp` файл се разглежда отделно от останалите по време на компилация. Всеки .cpp файл също така в този контекст се нарича **translation unit**.

Тоест компилаторът не вижда „цялата програма наведнъж“, а работи файл по файл.

---

## Linking

След като всички `.cpp` файлове са компилирани, идва ред на **linker-а**.

Неговата задача е:

- да свърже обектните файлове помежду им
- да намери дефинициите на използваните функции, променливи и класове
- да свърже нужните библиотеки
- да създаде краен артефакт

В резултат най-често получаваме:

- **изпълнима програма** (`.exe`);
- **статична библиотека** (`.lib`, `.a`);
- **динамична библиотека** (`.dll`, `.so`).

### Пример за linking грешка: липсваща дефиниция

```cpp
// main.cpp
int sum(int, int);

int main() {
    return sum(3, 4);
}
```

Тук компилаторът е доволен: знае, че съществува функция `sum`, защото има декларация.

Но ако никъде няма дефиниция на `sum`, при линкване ще получим грешка, защото линкерът не може да намери реализацията.

### Пример за linker грешка: multiple definition

Ако дефинираме една и съща глобална функция в два различни `.cpp` файла, линкерът ще види две дефиниции на един и същ символ и ще даде грешка.

---

## Compile error срещу linker error

Тази разлика е много важна:

- **compile error** — проблем в конкретен `.cpp` файл
- **linker error** — проблем при свързването между различните обектни файлове

Примери:

- липсва `;` → compile error
- достъп до `private` член отвън → compile error
- функция е декларирана, но не е дефинирана никъде → linking error
- една и съща функция е дефинирана в два `.cpp` файла → linking error

## Разделна компилация

### Какво представлява

Една програма на C++ може да бъде разбита в множество изходни файлове (`.cpp`), които се **компилират независимо един от друг**. Това се нарича **разделна компилация**.

### Защо ни е нужна

Разделната компилация ни позволява:

- да разделяме програмата на по-малки и логически обособени части
- да държим интерфейса отделно от реализацията
- да използваме един и същ код от много места
- да не прекомпилираме цялата програма при всяка промяна
- да работим по-лесно в екип

---

## Компилационна единица (translation unit)

Много полезен термин тук е **translation unit**.

> Translation unit е резултатът след preprocess-ването на един `.cpp` файл заедно с всичко, което е включено чрез `#include` в него.

Това е важно, защото:

- обяснява защо include guards работят;
- обяснява защо не include-ваме `.cpp` файлове;
- обяснява какво означава нещо да е видимо само в даден `.cpp`.

---

## Декларации и дефиниции

За да разберем добре разделната компилация, трябва да различаваме:

- **декларация** — казва, че нещо съществува;
- **дефиниция** — създава самото нещо или дава реализация.

### Пример с функция

```cpp
int sum(int a, int b); // декларация
```

```cpp
int sum(int a, int b) { // дефиниция
    return a + b;
}
```

### Защо това е важно

Компилаторът често има нужда само от **декларация**, за да знае как да използва нещо.

Линкерът обаче има нужда и от **дефиниция**, за да може накрая да свърже програмата успешно.

Header (`.h`) файловете обикновено съдържат **декларации**, а `.cpp` файловете — **дефиниции**.

---

## `.h` и `.cpp`

### Какво обикновено поставяме в `.h`

В header файловете обикновено поставяме:

- декларации на функции;
- декларации на класове;
- декларации на член-функции;
- типове, константи и други неща, които трябва да се виждат отвън.

### Какво обикновено поставяме в `.cpp`

В `.cpp` файловете обикновено поставяме:

- дефиниции на функции;
- дефиниции на член-функции;
- детайли по реализацията.

Можем да мислим за това така:

- `.h` файлът казва **какво** може да се използва;
- `.cpp` файлът показва **как** е реализирано.

---

## Защо не include-ваме `.cpp`

`#include` е текстова операция. Ако напишем:

```cpp
#include "something.cpp"
```

реално копираме съдържанието на този файл в текущия.

Това често води до:

- multiple definition грешки;
- объркване в структурата на проекта;
- нарушаване на идеята за разделна компилация.

Затова:

- include-ваме **header файлове**;
- не include-ваме **`.cpp` файлове**.

---

## Header guards

Понеже един header може да бъде включен повече от веднъж, трябва да се предпазим от многократно включване в рамките на един translation unit.

```cpp
#ifndef STUDENT_H
#define STUDENT_H

class Student {
public:
    void print() const;
};

#endif
```

### Как работят

- при първото включване `STUDENT_H` не е дефинирано;
- съдържанието на файла се включва и `STUDENT_H` се дефинира;
- при следващо включване в същия translation unit съдържанието се пропуска.

Така избягваме грешки от повторни декларации и дефиниции.

### `#pragma once`

Алтернатива на header guards е:

```cpp
#pragma once
```

Целта е същата — да не допускаме многократно включване на един и същ header.

---

## Пример за разделна компилация

### `Student.h`

```cpp
#ifndef STUDENT_H
#define STUDENT_H

class Student {
public:
    Student(const char* name, int age);
    ~Student();

    const char* getName() const;
    int getAge() const;
    void print() const;

private:
    char* name = nullptr;
    int age;
};

#endif
```

### `Student.cpp`

```cpp
#include "Student.h"
#include <iostream>
#include <cstring>

Student::Student(const char* name, int age) : age(age) {
    this->name = new char[std::strlen(name) + 1];
    std::strcpy(this->name, name);
}

Student::~Student() {
    delete[] name;
    name = nullptr;
}

const char* Student::getName() const {
    return name;
}

int Student::getAge() const {
    return age;
}

void Student::print() const {
    std::cout << name << " (" << age << ")\n";
}
```

### `main.cpp`

```cpp
#include "Student.h"

int main() {
    Student s("Ivan", 20);
    s.print();
}
```

Тук:

- `main.cpp` вижда само интерфейса на класа;
- реализацията е в `Student.cpp`;
- двата `.cpp` файла се компилират отделно;
- линкерът ги свързва в една програма.

## Връзка между разделната компилация и `static`

След като програмата вече е разделена в няколко `.cpp` файла, възникват няколко въпроса:

- кои имена са видими от други файлове
- кои имена трябва да останат локални за даден `.cpp`
- кои променливи трябва да живеят през цялото изпълнение на програмата
- кои данни принадлежат на конкретен обект и кои на целия клас

Точно тук идва ролята на `static`.

Важно е да помним, че думата `static` се използва в **различни контексти** и няма едно-единствено значение.

---

## `static` функции и променливи (на ниво файл)

Когато използваме `static` при свободна функция или глобална променлива в даден `.cpp` файл, тя става видима **само в текущия translation unit**.

Това е силно свързано с темата за разделната компилация и линкването.

### Пример без `static`

`a.cpp`

```cpp
#include <iostream>

void helper() {
    std::cout << "from a.cpp\n";
}
```

`b.cpp`

```cpp
#include <iostream>

void helper() {
    std::cout << "from b.cpp\n";
}
```

Ако и двата файла участват в една и съща програма, линкерът ще види две дефиниции на една и съща глобална функция и най-вероятно ще даде грешка.

### Пример със `static`

`a.cpp`

```cpp
#include <iostream>

static void helper() {
    std::cout << "from a.cpp\n";
}
```

`b.cpp`

```cpp
#include <iostream>

static void helper() {
    std::cout << "from b.cpp\n";
}
```

Сега всяка функция е видима само в собствения си `.cpp` файл.

### Идеята

Тук `static` означава:

> Tова име да не се вижда извън текущата компилационна единица (translation unit).

Същата идея важи и за глобални статични променливи на ниво файл.

---

### `extern` и глобални променливи в няколко файла

Понякога искаме една глобална променлива да бъде **дефинирана в един `.cpp` файл**, но да може да се използва и от други `.cpp` файлове.

Това се прави чрез `extern`.

- **дефиниция**: създава самата променлива
- **декларация с `extern`**: казва, че променливата съществува някъде другаде

#### Пример

#### `globals.h`
```cpp
#ifndef GLOBALS_H
#define GLOBALS_H

extern int x;

#endif
```

#### `globals.cpp`
```cpp
#include "globals.h"

int x = 42;
```

#### `main.cpp`
```cpp
#include <iostream>
#include "globals.h"

int main() {
    std::cout << x << '\n';
    x = 100;
}
```

#### `other.cpp`
```cpp
#include <iostream>
#include "globals.h"

void printX() {
    std::cout << x << '\n';
}
```

- `int x = 42;` е **дефиницията** на променливата
- `extern int x;` е **декларация**, че такава променлива съществува, но е дефинирана другаде
- всеки `.cpp`, който include-не `globals.h`, може да използва `x`
- при линкване всички използвания на `x` се свързват с единствената ѝ дефиниция в `globals.cpp`

#### Разлика със `static`

Ако вместо това напишем:

```cpp
static int x = 42;
```

в `globals.cpp`, тогава `x` ще бъде видима **само в този translation unit** и няма да може да се използва от други `.cpp` файлове.

Следователно:

- `static` на ниво файл → скрива името в рамките на текущия `.cpp`
- `extern` → позволява да използваме глобална променлива, дефинирана в друг `.cpp`

> Добра практика е глобалните променливи да се използват внимателно, защото въвеждат споделено състояние в програмата.

## Статични локални променливи

Можем да правим статични локални променливи в рамките на някакъв scope (в случая най-често става въпрос за локални за функции статични променливи). Ако направим локална за функция променлива `static`, тя запазва стойността си между различните извиквания на функцията.

```cpp
#include <iostream>

int increment() {
    static int value = 0;
    ++value;
    return value;
}

int main() {
    std::cout << increment() << std::endl; // 1
    std::cout << increment() << std::endl; // 2
    std::cout << increment() << std::endl; // 3
}
```

### Какво е важно тук

- **scope** — променливата е видима само в тялото на функцията;
- **lifetime** — променливата живее до края на програмата;
- **инициализация** — извършва се само веднъж, при първото извикване на функцията!

Това е добър пример, че в C++ трябва да различаваме:

- къде едно име е видимо;
- колко дълго живее съответният обект.

### Кога е полезно

- брояч на извиквания;
- проста кешираща логика;
- запазване на състояние между извиквания.
- *Singleton Design Pattern*

## Статична член-данна на клас

Статичната член-данна **не** принадлежи на конкретен обект, а на *целия клас*.

Това означава, че всички обекти от класа **споделят** **една и съща стойност**.

### Пример

```cpp
class Student {
private:
    static int createdCount;

public:
    Student();
    static int getCreatedCount();
};
```

В класа имаме **декларация** на статичната член-данна. Тя трябва да бъде **дефинирана** извън класа:

```cpp
int Student::createdCount = 0;
```

Ако пропуснем тази дефиниция и използваме `createdCount`, най-често ще получим linker грешка.

### Пълен пример

```cpp
class Student {
private:
    static int createdCount;

public:
    Student() {
        ++createdCount;
    }

    static int getCreatedCount() {
        return createdCount;
    }
};

int Student::createdCount = 0;
```

Така можем да броим колко обекта са били създадени.

### Какво трябва да помним

- статичната член-данна е една за целия клас
- тя не е част от всеки конкретен обект
- обикновено се дефинира в `.cpp` файла

---

## Статична член-функция на клас

Статичната член-функция, подобно на статичните член-данни, също не е свързана с конкретен обект, а с целия клас.

### Характеристики

- няма указател `this`!
- не може директно да достъпва нестатични член-данни
- може директно да достъпва единствено статични член-данни
- извиква се чрез името на класа

### Пример

```cpp
class Test {
private:
    static int x;

public:
    static void f() {
        x *= 2;
    }
};

int Test::x = 1;

int main() {
    Test::f();
}
```

### Защо няма достъп до нестатичните членове

Нестатичната член-функция работи върху конкретен обект и затова има `this`.

Статичната член-функция не работи върху конкретен обект, затова няма `this` и не може директно да използва нестатични членове.

Пример:

```cpp
class A {
public:
    static void f();
    int x;
};

void A::f() {
    // x = 5; // грешка
}
```

### Добра практика

Когато използваме static членове на класа, е по-коректно да ги достъпваме чрез името на класа:

```cpp
Test::f();
```

а не чрез конкретен обект.

## Различни употреби на `static` (обобщение)

Важно е да не смесваме различните му употреби.

В зависимост от контекста `static` означава различни неща:

- при функция или глобална променлива на ниво файл  
  → името е видимо само в текущия `.cpp`;

- при локална променлива (във scope-a на функция)  
  → стойността се запазва между извикванията;

- при член-данна на клас  
  → данната е обща за всички обекти;

- при член-функция на клас  
  → функцията принадлежи на класа, а не на конкретен обект.

## Кратка бележка за modules и CMake

### C++ modules

Съвременният C++ предлага и **modules** като по-модерен механизъм за модулност.

Те имат за цел да решават част от проблемите на традиционните header-и, но засега в курса основният модел, който трябва да владеем добре, остава:

- `.h` за декларации
- `.cpp` за дефиниции

### CMake

В по-големи проекти често използваме **build systems**, които автоматизират компилацията и линкването. Един от най-разпространените инструменти е **CMake**.

На практика:

- CMake описва как се компилира проектът;
- може да работи и на Windows;
- има поддръжка и във Visual Studio.

За нашите цели засега е по-важно да разберем добре какво реално се случва при компилация и линкване, преди да използваме инструмент, който го автоматизира.

## Най-важното от темата

### За компилацията

- препроцесорът обработва директивите, започващи с `#`;
- всеки `.cpp` файл се компилира отделно;
- в резултат получаваме обектни файлове;
- линкерът ги свързва в крайна програма или библиотека.

### За разделната компилация

- големите програми се разделят на множество файлове;
- `.h` файловете съдържат основно декларации;
- `.cpp` файловете съдържат основно дефиниции;
- **не** include-ваме `.cpp` файлове
- header guards предотвратяват многократно включване.

### За `static`

- `static` няма едно-единствено значение;
- при функции/променливи на ниво файл ограничава видимостта до текущия `.cpp`;
- при локални променливи запазва стойността между извикванията;
- при член-данни ги прави общи за целия клас;
- при член-функции ги прави независими от конкретен обект.
